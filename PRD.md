# PRD — 오늘의 심부름 주문

## 1. 서비스 개요

| 항목 | 내용 |
|---|---|
| 서비스 이름 | 오늘의 심부름 주문 |
| 한 줄 소개 | 오늘 판매하는 심부름 메뉴를 보여주고, 회원가입한 손님이 메뉴를 담아 주문하면 손님은 "내 주문"에서 자기 주문 상태를, 사장은 관리 화면에서 전체 주문을 확인하는 서비스 |
| 타깃 사용자 | 심부름을 파는 초등학생(사장) + 심부름을 시키는 손님(가족/이웃 등) |
| 컨셉 | 용돈벌이를 위한 미니 주문 접수 서비스. 결제는 다루지 않고 "주문 접수 → 완료 처리" 흐름만 다룸 |
| 대표 문구 | 작은 손이 하는 큰 심부름 |

## 2. 핵심 기능 (6가지)

| # | 기능 | CRUD | 주체 | 설명 |
|---|---|---|---|---|
| 1 | 메뉴 보여주기 | Read (정적 데이터) | 누구나 | 오늘의 심부름 메뉴 카드 목록, 로그인 불필요 |
| 2 | 회원가입·로그인 | — (Auth) | 손님/사장 | 이메일 기반 가입/로그인 (Supabase Auth) |
| 3 | 주문 담기 & 접수 | Create | 손님 (로그인) | 메뉴를 장바구니에 담아 주문 저장 |
| 4 | 내 주문 보기 | Read (본인 것만) | 손님 (로그인) | 자신이 남긴 주문과 상태만 조회 |
| 5 | 사장 전체 조회 & 완료 처리 | Read / Update | 사장 | 전체 주문 조회, 상태를 "접수 → 완료"로 변경 |
| 6 | 주문 삭제 | Delete | 사장 | 장난·취소 주문 삭제 |

## 3. 화면/라우트 명세

### 3.1 주문 페이지 — `/` (`src/pages/OrderPage.jsx`)
- **상단**: 가게 이름 + 장바구니 배지
  - 비로그인: `[로그인]` 버튼
  - 로그인: `[내 주문]` `[로그아웃]` 버튼
- **중간**: 오늘의 메뉴 카드 목록 — `MenuCard` 컴포넌트 반복 렌더링
- **하단**: 주문서
  - 담은 품목 목록 (수량 `+`/`−` 조절)
  - 합계 금액
  - 심부름 희망 시간 입력
  - `[주문하기]` 버튼 (비로그인 상태에서는 `[로그인하고 주문하기]`로 대체 → 클릭 시 `/auth`로 이동)

### 3.2 회원가입·로그인 — `/auth` (`src/pages/AuthPage.jsx`)
- shadcn `Tabs` 컴포넌트로 `[로그인]` / `[회원가입]` 전환
- 입력 필드: 이메일, 비밀번호

### 3.3 내 주문 — `/my` (`src/pages/MyOrdersPage.jsx`)
- 로그인 필요 (비로그인 접근 시 `/auth`로 리다이렉트)
- 로그인한 손님 본인의 주문만 최신순 목록
- 표시 항목: 품목, 금액, 상태(Badge), 주문 시각

### 3.4 관리 화면 — `/admin` (`src/pages/AdminPage.jsx`)
- 비로그인 또는 손님(사장 아님) 접근 시 → `/`로 리다이렉트 (화면에 노출 자체를 막음)
  - 단, 이 리다이렉트는 **UX 편의일 뿐** 실제 데이터 보호 수단이 아님 — 진짜 접근 제어는 5장의 RLS 정책이 담당 (자세한 내용은 5장 참고)
- 전체 주문 표: 주문자, 품목, 금액, 심부름 희망 시간, 상태, 주문 시각
- 각 행에 `[완료]` `[삭제]` 버튼
- `[로그아웃]` 버튼

## 4. 데이터 모델

### 4.1 `orders` 테이블

| 컬럼 | 타입 | 설명 | 기본값 |
|---|---|---|---|
| `id` | uuid | 기본키 | `gen_random_uuid()` |
| `user_id` | uuid (FK → auth.users) | 주문자, 로그인 계정과 자동 연결 | — |
| `items_text` | text | 주문 내역 (예: `"어깨안마 x2, 설거지 x1"`) | — |
| `total_price` | integer | 합계 금액 | — |
| `pickup_time` | text | 심부름 희망 시간 | — |
| `status` | text | 주문 상태 | `'접수'` |
| `created_at` | timestamptz | 주문 시각 | `now()` |

> 상태값은 `'접수'` / `'완료'` 두 가지만 사용 (범위 내 요구사항 기준).

### 4.2 메뉴 데이터 — `src/data/menu.js`
DB가 아니라 코드에 정적 배열로 관리 (등록/수정 화면 없음, 사장이 파일 직접 수정).

```js
// src/data/menu.js
export const menu = [
  { id: 1, name: "어깨안마", price: 500, description: "5분 어깨안마" },
  { id: 2, name: "설거지", price: 1000, description: "저녁 설거지 대행" },
  // ...
];
```

### 4.3 `profiles` 테이블 (사장 판별용)

역할(손님/사장) 구분을 위한 최소 테이블. 새 기능이 아니라, 5장의 "사장은 RLS로 권한 부여"를 실제로 구현하기 위한 필수 데이터임.

| 컬럼 | 타입 | 설명 | 기본값 |
|---|---|---|---|
| `id` | uuid (FK → auth.users, PK) | 계정과 1:1 매칭 | — |
| `role` | text | `'손님'` 또는 `'사장'` | `'손님'` |

- 회원가입 시 트리거로 `profiles` 행을 자동 생성하고 `role`은 항상 `'손님'`으로 시작한다.
- 특정 계정을 사장으로 만드는 절차는 **Supabase 콘솔(또는 SQL Editor)에서 운영자가 직접** 해당 행의 `role`을 `'사장'`으로 수정하는 것뿐이다. 앱 화면에는 이 값을 바꾸는 API/버튼이 전혀 없다.

## 5. 회원·권한 설계

| 역할 | 가입 방법 | 할 수 있는 일 |
|---|---|---|
| 방문자 (비로그인) | — | 메뉴 보기만 |
| 손님 (로그인) | `/auth` 회원가입 화면에서 직접 | 주문 남기기 + **자기** 주문만 조회 |
| 사장 (로그인) | 가입 방법은 손님과 동일, 권한은 RLS 정책으로 부여 | 전체 조회 · 완료 처리 · 삭제 |

- 손님 A는 손님 B의 주문을 절대 볼 수 없다 — **화면 로직이 아니라 Postgres RLS(Row Level Security) 정책이 보장**한다.
- "사장 가입" 별도 버튼은 없음. 사장 지정은 화면 밖(Supabase 콘솔)에서 4.3 `profiles.role`을 직접 수정하는 방식으로만 처리.
- **원칙: 사장 권한은 화면(프론트엔드) 코드가 아니라 DB의 RLS 정책이 최종 판단한다.** `/admin` 라우트 가드나 버튼 노출 여부는 사용자 경험을 위한 것일 뿐, 로그인한 손님이 개발자 도구로 Supabase API를 직접 호출해도 아래 정책에 의해 실제로 막힌다.
- 사장 판별 함수 (RLS에서 재사용, `profiles` 테이블을 직접 참조하는 재귀 문제를 피하기 위해 `SECURITY DEFINER` 함수로 분리):

  ```sql
  create or replace function public.is_owner()
  returns boolean
  language sql security definer
  as $$
    select exists (
      select 1 from public.profiles
      where id = auth.uid() and role = '사장'
    );
  $$;
  ```

- `orders` 테이블 RLS 정책 (구현 단계에서 SQL로 작성 예정):
  - `select`: `user_id = auth.uid()` (본인 주문) **OR** `is_owner()` — 이 조건이 손님 A/B 간 주문 격리와 사장의 전체 조회를 동시에 보장하는 유일한 지점
  - `insert`: `with check (user_id = auth.uid() and status = '접수')` — 본인 명의로만 생성 가능 + 접수 이외 상태로 위장 생성하는 것을 차단 (완료 처리는 오직 update 정책을 통해서만 가능)
  - `update`: `using (is_owner()) with check (is_owner())` — 사장만 상태 변경 가능
  - `delete`: `using (is_owner())` — 사장만 삭제 가능
- `profiles` 테이블 자체의 RLS: 본인 행 **조회만** 허용하고, `role` 컬럼에 대한 **update 정책은 두지 않는다** (일반 사용자용 UPDATE 정책 없음). 만약 손님이 자기 `role`을 스스로 `'사장'`으로 바꿀 수 있는 경로가 하나라도 열려 있으면 위 모든 RLS가 무력화되므로, 이 부분이 권한 설계 전체의 최종 방어선이다.
